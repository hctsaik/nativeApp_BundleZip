Annotation Platform Architecture Discussion

Multi-agent 設計整理：評估目前系統是否足以彈性擴展多種 labeling tool、常見 annotation format， 以及如何讓不同 Oracle schema、Oracle table、web service、權限與資料欄位的外部系統容易接入。

70% Format adapter 彈性已有基礎
45% Labeling tool runtime 還需要 adapter 化
30% Enterprise integration 尚缺正式 common interface

1. Executive Summary

判斷： 目前系統已經從 X-AnyLabeling 專用，演進到支援 LabelMe、X-AnyLabeling、ISAT、COCO、 YOLO detection、YOLO segmentation 的第一版多格式架構。核心方向正確：平台內部保留 canonical annotation model，外部格式透過 adapter 進出。

但如果目標是讓不同系統容易對接，目前還不算完整彈性。主要缺口不是再多加一個 exporter， 而是需要把 Format Registry、Labeling Tool Adapter、 External System Connector、Schema Mapping Profile、 Capability Matrix 與 Conversion Loss Report 拉成正式 common interface。

最重要的設計原則是：annotation core 永遠只處理平台 canonical model；所有外部格式差異放在 format adapter；所有工具差異放在 tool adapter；所有客戶系統差異放在 integration connector 與 mapping profile。這樣才能同時支援 LabelMe、ISAT、COCO、YOLO，也能接不同 Oracle schema、 REST service、身份權限與資料欄位，而不讓核心流程被客製邏輯污染。

可行性結論： 這個 solution 是可行的，但應採漸進式改造。第一步不要直接做大型 plugin system 或 Oracle 雙向同步，而是先把目前已存在的 adapters、tool launch、service dispatch 收斂到 registry 與 contract。 等核心 extension point 穩定後，再把外部系統 connector、mapping profile、export delivery 接上去。

2. Feasibility Review

重新檢視後，我認為這份設計的方向正確，且可以落在目前 repo 架構上。它不是要推翻既有 AnnotationService、MCP、module_006/008/009/012，而是把這些既有入口逐步變成 registry 與 connector 的 façade。

設計項目 可行性 落地方式 主要風險
FormatRegistry 高 把目前 LabelMe、X-AnyLabeling、ISAT、COCO、YOLO adapters 先註冊成 descriptors，再讓 service 透過 registry dispatch。 若 descriptor 太複雜，第一版會拖慢開發；應先支援 import/export/capability/aliases。
ToolRegistry 中高 包住現有 labeling_runtime.py 與 X-AnyLabeling runtime，逐步讓 module_006/012 使用同一套 launch contract。 不同 GUI 工具的 CLI 能力不一致，collect results 需要以 capability 表示，不要假設每個工具都能完整自動化。
ExternalSystemConnector 中 先做 contract、fake connector、file connector，再接 Oracle/REST。核心不要直接 import Oracle driver。 客戶系統差異大，若沒有 profile/mapping versioning，會變成到處加客製 if/else。
IntegrationProfile 高 用 JSON/YAML 描述 system_id、tenant、connector、field mapping、schema mapping、format policy。 profile 欄位要版本化，否則後續 migration 會痛。
Structured ConversionReport 高 延伸既有 ConversionReport，加入 loss summary、field losses、mapping version、dry-run 結果。 若 UI 不顯示 loss report，使用者仍會誤以為 export 一定無損。
Module migration 中 module_006 已接近 generic；module_008 需從 LabelMe-style session JSON 轉向 canonical bridge；module_009 需分階段拆 X-Any coupling。 module_009 tracking correction 與 X-AnyLabeling 結合較深，不適合一次改完。
建議落地策略： 先把 registry 與 contract 建起來，保留既有 public API 與 legacy aliases。也就是外部呼叫者仍可用 annotation_create_export、annotation_prepare_labeling_project， 但內部 dispatch 從 hard-coded if/elif 改成 registry。這樣風險最低，也最容易用現有 tests 證明沒有 regression。

4. Multi-Agent Findings

Agent A：目前 repo 架構

sidecar/python-engine/annotation 已有 canonical model、format adapters、service API； mcp/annotation_mcp 已有 generic tools；OpenSpec 也明確區分 annotation format 與 labeling tool。這是好的基底。

但 AnnotationService 仍用 if/elif 做 adapter dispatch； module_008 內部仍偏 LabelMe-style JSON； module_009 tracking correction 仍高度 X-AnyLabeling specific； module_012 自己做 tool launch dispatch。這些都是下一步要收斂的地方。

Agent B：外部系統整合

不同系統的差異不只在標註格式，也包含 Oracle table/schema、web service API、身份權限、 資料 lifecycle、審核流程、回寫策略、錯誤重試與版本衝突。這些不應該進 annotation core， 應由 integration connector 與 mapping profile 承接。

Agent C：Format / Export / Import

LabelMe、X-AnyLabeling、ISAT、COCO、YOLO 的資料模型不同。困難點不是能不能寫出檔案， 而是轉換時哪些資訊會遺失。例如 polygon 匯出 YOLO detection、bbox 匯出 YOLO segmentation、 mask 匯出未支援格式，都應在 conversion report 中清楚呈現。

6. Tool 與 Format 必須分開

設計上要明確分開兩個概念：

  • Labeling tool：可啟動或準備 project 的工具，例如 X-AnyLabeling、LabelMe、ISAT。
  • Annotation format：檔案與資料交換格式，例如 LabelMe JSON、ISAT JSON、COCO、YOLO detection、YOLO segmentation。
一個工具可能使用某種格式，但不等於格式本身。例如 X-AnyLabeling 大致使用 LabelMe-compatible JSON， 但有自己的 launch args、project layout、rectangle 表現、工具行為。Export 時使用者選 COCO 或 YOLO， 不代表標註時一定要用 COCO 或 YOLO 工具。

10. Export 應如何決定 LabelMe / ISAT / COCO / YOLO

Export 不應只看使用者標註時用了哪個工具，而應由三個因素共同決定：

  • 用途：human review、training、customer handoff、model evaluation、archive。
  • 目標系統能力：客戶系統能吃 COCO、YOLO、ISAT、LabelMe 或自訂格式。
  • 資料內容：bbox、polygon、mask、classification、track、attribute 是否會在目標格式中失真。

建議 export API 使用明確 request： 

{
  "annotation_set_id": "aset_xxx",
  "target_system": "customer_a_mes",
  "purpose": "training",
  "export_format": "coco",
  "delivery": {
    "type": "connector_push",
    "sink": "customer_a_oracle_annotation_sink"
  },
  "options": {
    "include_images": false,
    "class_id_policy": "schema_order",
    "on_loss": "warn"
  }
}
若使用者沒有指定 export_format，平台可以從 integration profile 的 format_policy.default_export_format 選擇。但實際匯出前應做 dry-run capability check， 讓使用者知道是否會丟失 polygon、attribute、mask、instance metadata。

11. Import / Export 流程建議

Export flow

使用者或外部系統建立 export request
  -> 查 IntegrationProfile 與 FormatRegistry
  -> 檢查 CapabilityMatrix
  -> 套用 SchemaMappingProfile
  -> Dry-run conversion
  -> 產生 LossReport preview
  -> 使用者或 policy 確認
  -> 實際 export artifact
  -> 儲存 artifact + conversion report + mapping profile version
  -> 透過 AnnotationSinkConnector 回寫外部系統

Import flow

使用者或外部系統指定 import format
  -> Registry 找 adapter
  -> 解析檔案或資料夾
  -> AssetResolver 對應 image
  -> SchemaMappingProfile 對應 label/category/class index
  -> Geometry normalization
  -> Validation
  -> 寫入 canonical AnnotationSet
  -> 回傳 unmatched / skipped / approximated / unknown labels report

13. 不污染 Annotation Core 的做法

  • Core model 只放平台共通欄位，不放 CUSTOMER_A_IMAGE_ID 這種欄位。
  • 外部 primary key 放在 metadata 或 provenance.external_refs。
  • Oracle SQL、REST URL、token、table name 只存在 connector config。
  • 客戶欄位轉換寫在 mapping profile，不寫在 annotation service。
  • 客戶特殊 JSON 寫成 custom format adapter，不修改既有 LabelMe/COCO adapter。
  • 客戶特殊 push 行為寫成 sink connector，不修改 core storage。
  • 多 tenant 時，workspace、schema、credential、audit log 應有清楚 boundary。

15. Test Strategy

Contract tests

  • 每個 format adapter 必須能列出 descriptor/capability。
  • 每個 adapter 都能 export empty annotation set。
  • bbox、polygon、classification 支援子集要有共同測試。
  • 遇到 unsupported geometry 不可 silent fail，必須寫 loss report。
  • import 後 canonical model 正確。
  • class mapping deterministic。

Golden file tests

為 LabelMe、ISAT、COCO、YOLO、Pascal VOC、CVAT 建立小型 golden samples。 每次 adapter 修改都比較輸出 JSON/XML/TXT 結構。

Round-trip tests

lossless format:
  canonical -> format -> canonical

lossy format:
  canonical -> format -> canonical + expected loss report

Integration contract tests

  • Oracle connector 用 fake Oracle 或 contract fixture 測 list/pull/push。
  • REST connector 用 mock server 測 auth、pagination、retry、conflict。
  • mapping profile 測未知 label、版本差異、欄位缺失。

17. Final Recommendation

建議： 你現在的系統已經有可擴展多格式 labeling 的基礎，但要成為能讓不同系統穩定對接的平台， 下一步應該不是繼續在 service 裡加更多 if/elif，而是正式建立 common interface： FormatRegistry、ToolRegistry、ExternalSystemConnector、 IntegrationProfile、SchemaMappingProfile 與 structured ConversionReport。

做完這一層之後，未來新增 Label Studio、CVAT、Pascal VOC、客戶自訂 Oracle schema、REST web service、 或自訂 export JSON，都應該是「新增 adapter/profile/connector」，而不是改 annotation core。

18. 實作設計：如何落地

下面是建議的實作順序。重點是每一階段都能獨立測試、保留既有 API、避免一次改壞既有 workflow。

Phase 0：建立 OpenSpec 與 contract acceptance criteria

  • 新增 openspec/changes/annotation-format-registry-and-external-integration/。
  • 定義四組需求：format registry、tool registry、integration profile、external connector。
  • 把「不修改 annotation core 來支援客戶系統」寫成明確 requirement。
  • 把 export dry-run、loss report、mapping profile version 寫成 acceptance criteria。

Phase 1：FormatRegistry 先落地，替換 service if/elif

先新增格式 contract，但不要搬動既有 adapter 的檔案位置，降低風險。

  contracts.py       # FormatAdapter, FormatDescriptor, FormatCapabilities
  registry.py        # register(), get(), normalize(), list_supported()
  builtins.py        # 註冊 labelme/x-anylabeling/isat/coco/yolo adapters
  • 將現有 labelme.py、isat.py、coco.py、yolo_detection.py、yolo_segmentation.py 包成 adapter descriptor。
  • AnnotationService.supported_annotation_formats() 改讀 registry。
  • AnnotationService.create_export()、import_annotations()、import_project_labels() 改由 registry dispatch。
  • 保留 import_xanylabeling_* 與既有 MCP tools，內部改呼叫 registry。

Phase 2：Structured ConversionReport 與 export dry-run

21. 外部整合設計疑問與建議（深度評論）

以下為從「外部系統整合工程師」角度，針對設計文件提出的具體疑問。每條都附上風險與建議。

21.1 ExternalSystemConnector 介面疑問

Q1：list_tasks 的分頁機制 — offset-based vs cursor-based
目前介面簽名 list_tasks(query, page) 隱含 offset-based pagination。 Oracle 大量資料時，offset-based 有嚴重 skew：若 import 進行中有新任務插入，同一筆可能被重複或跳過。 更常見的是 DBA 提供 View，無 cursor token 概念。
建議： page 改為 PaginationToken（opaque string），各 Connector 自行決定實作方式（keyset / stored procedure cursor / offset）。需明確規定「任務列表的穩定排序欄位」作為前提條件。
Q2：resolve_asset — 影像在 Oracle 中以 BLOB 存放時如何處理？
設計假設影像有可直接存取的 file path，但企業環境常見 BLOB 欄位、SMB share 無法掛載、或 Token-based 短效 URL。 若無 streaming/temp download 機制，記憶體壓力大（工業相機影像動輒 50MB+），且暫存檔若無清理會洩漏磁碟空間。
建議： ResolvedAsset 改為 Union type，支援 LocalFilePath | RemoteURL | BlobStream | ContentAddressedRef，含 ttl 欄位（Token URL 過期時間）。Connector 負責 BLOB 下載，暫存路徑由平台統一注入。
Q3：check_conflicts — conflict 語意完全未定義
並發寫入衝突、版本衝突、語意衝突（label 不一致）是三種完全不同的問題。 用 check-then-act 模式防並發衝突是 TOCTOU 問題，根本無法保護。
建議： 明確定義 Conflict type system（含 conflict_type 和 resolution_hint），並發保護由 DB-level locking 處理，check_conflicts 只做版本號對比查詢。
Q4：push_annotations 的 mode 語意不完整
insert-only / upsert / replace-all / append-labels / patch 行為差異巨大。 Upsert 的 natural key（「以哪個 key 判斷已存在」）完全未定義。
建議： mode 改為結構化物件，Connector 在 CapabilityMatrix 中聲明支援哪些 mode，job 建立時就 fail-fast，不要等 push 時才報錯。
Q5：Oracle Batch Commit — all-or-nothing 支援
若外部系統要求一個 task 下所有 annotation 必須 all-or-nothing（同時寫 header + detail 兩張表）， Orchestration 層如果逐筆調用 push，中途失敗會造成 Oracle 資料不一致。 Retry 若無冪等 key，可能造成重複寫入。
建議： Connector 需聲明 supports_batch_commit 和 idempotency_key_field，Orchestration 層尊重 Connector 的 transaction boundary 宣告。
Q6：錯誤重試策略缺失
哪些錯誤是 Transient（retry with backoff）、哪些是 Fatal（dead-letter + 告警）、哪些是 BusinessLogic（人工介入）完全未定義。
建議： Exception 要帶分類標記，Orchestration 根據標記決定處理路徑；沒有 dead-letter 機制的話，失敗 job 可能靜默消失。

21.2 IntegrationProfile 設計疑問

Q7：Profile 缺乏版本欄位 — Audit 追溯失效
如果 field_mapping 被修改，過去的 export artifact 無法追溯是用哪個版本 mapping 產生。 Regulatory 環境（品管記錄）可能要求完整重現能力。
建議： Profile 必須有 version + effective_from + checksum；每次 export job 的 provenance 記錄必須儲存「執行時的 profile snapshot」，而非只是參照可能被修改的 profile ID。
Q8：query_id 安全性 — SQL Injection 風險
"query_id": "pending_inspection_images" 若直接代入 SQL 執行，存在 SQL injection 風險。 即使是內部系統，若 profile 被惡意修改，後果嚴重。
建議： query_id 必須對應 allowlist registry（管理員預先登記合法的 query handle），Connector 只能引用已登記的 handle。
Q9：Credential 管理完全缺失（最高優先）
Profile 裡完全沒有 Oracle connection string、密碼、REST API token 的存放設計。 若寫在 profile JSON 中，任何能讀取 profile 的人都能看到明文 credential。
建議： 獨立 CredentialStore 機制，profile 只存 secret reference（"credential_ref": "vault://fab_01/oracle_prod"），Connector 透過 CredentialProvider 介面取得認證，不從 profile 直接讀取。這是實作第一個 Oracle Connector 之前必須設計完成的前提條件。
Q10：同一客戶多個 Oracle Instance（prod/staging）無法區分
system_id: "customer_a_mes" 無法區分 prod/staging，但兩個環境的 field_mapping 往往不同（staging schema 常落後一個版本）。
建議： Profile 加入 environment 欄位，執行 job 時強制校驗 annotation platform 自身環境與 profile target environment 一致，否則拒絕執行。
Q11：Schema Source 不可用時的 Fallback 機制缺失
schema_source: {type: "web_service"} 若 web service 暫時維護中，整個標注 workflow 是否全面停擺？ 若 schema 動態變化（今天與昨天不同），進行中的標注使用過期 label 集合。
建議： 必須定義 schema TTL、不可用時的 fallback 行為（用 cached version 繼續 vs. fail），以及 schema immutability 保證：AnnotationSet 一旦開始使用某版 schema 就 pin 住，直到完成。

21.3 Schema Mapping 疑問

Q12：warn_and_skip 的 skip 粒度未定義
跳過「這個 annotation」vs 跳過「整張圖所有 annotation」vs 跳過「整個 batch job」，損失範圍差異極大。 warn 的 warning 要送到哪裡？沒有追蹤機制的 warning 等於沒有。
建議： 明確定義 skip 粒度，並且必須產生結構化 SkipReport（含每個被跳過的 annotation ID + 原因），讓使用者在 confirm export 之前就能看到，而非事後從 log 挖。
Q13：樹狀 Label Taxonomy 的 Mapping 無法表達
Flat key-value mapping 無法處理「缺陷 > 外觀缺陷 > 刮痕」這樣的階層 label。 Export 時父節點 category 資訊會丟失，外部系統若期望 hierarchical 結構則無法填充。
建議： Schema Mapping 支援 path-based mapping（"缺陷/外觀缺陷/刮痕": {"external_class": "SCRATCH", "external_parent": "SURFACE"}），CapabilityMatrix 加入「是否支援 hierarchical label」維度。
Q14：Label 為數字 ID 時的型別轉換問題
JSON key 必然是字串，Oracle 中 DEFECT_CLASS_ID = 7（integer）vs "7"（string）可能靜默 mapping 失敗，尤其是有 leading zero 或 compound key 時。
建議： Profile 加入 label_id_type: "string" | "integer" | "compound"，Connector 層做明確型別轉換。
Q15：Schema Mapping 版本更新 — 歷史 AnnotationSet 如何處理
客戶把 "DEFECT_A" 廢棄、改為 "DEFECT_A1"/"DEFECT_A2" 時， 已完成的 AnnotationSet 要 retroactive 更新？freeze？dual-label？這是業務決策，不能留到實作時再定。
建議： 必須在設計階段確定 migration 策略，且 AnnotationSet 必須有 schema_version_ref 欄位，SchemaStore 必須是 immutable（不能刪除已被引用的 schema 版本）。

21.4 多 Tenant 與 Audit

Q16：Canonical Model 是否有 tenant_id？資料隔離層級未定義
設計文件提到 multi-tenant 但 canonical model 欄位沒有列出 tenant_id。 若靠 application logic 做隔離（WHERE clause 加 tenant filter），一個遺漏的 query 就可能造成跨 tenant 資料洩漏。
建議： 必須明確決定 tenancy model：Row-level isolation（tenant_id + RLS）、Schema-level isolation（每 tenant 獨立 DB schema），或 Database-level isolation。這個決策影響所有 CRUD 操作設計，必須在第一行 DB migration 之前確定。
Q17：Audit Log 設計完全缺失
誰在什麼時間用哪個 profile 版本對哪個 AnnotationSet 做了什麼操作（export/import/delete）、結果成功還是失敗——無從查詢。 Regulatory 環境（ISO、FDA 21 CFR Part 11）可能強制要求完整 audit trail。
建議： 獨立 AuditLog 機制，每個 job 完成（無論成功失敗）都寫入，且必須 append-only（任何 API 都不能刪除或修改）。

文件沒有說明第一次對接 Oracle 時的 onboarding 步驟、需向客戶 DBA 索取哪些資訊、需要哪些帳號權限，以及如何驗證 connector 設定正確。
建議： 設計 ConnectorHealthCheck 介面，讓 Management UI 可以逐步驗證：① 連線 ping、② list_tasks 冒煙測試、③ resolve_asset 可達性、④ push_annotations dry-run。詳見第 22 章 Diagram 3。
Q19：外部系統反向通知機制缺失
目前是單向：annotation platform 拉取任務、推送結果。若 MES 品管員人工審核後認為標注有誤，要如何通知平台重新標注？
建議： 定義 callback/webhook 機制，或在 list_tasks 查詢中包含「因 rework 被退回」的任務狀態，ExternalTask 加入 rework_reason 欄位。
Q20：Event-Driven 外部系統支援缺失
設計假設 annotation platform 主動 polling。若外部系統用 Kafka / RabbitMQ / Oracle AQ 發布事件，現有架構無法配合，且 polling 對 Oracle 造成查詢壓力。
建議： Connector 在 CapabilityMatrix 中聲明工作模式（pull vs push），Orchestration 根據 connector 宣告選擇驅動方式。
Q21：Orchestration 的觸發機制未定義
是 cron job、API call trigger、還是外部事件？沒定義的話各 Connector 可能各自實作 scheduler，造成競爭條件。
建議： 文件化觸發來源、最小觸發間隔、concurrent job 限制（同一 profile 同時只能有一個 running import job？），及 job 狀態機（queued → running → succeeded/failed/partial）。

21.6 三個最重要的實作前決策

決策一（最緊急）：Tenancy Model 與 Canonical Model 資料隔離層級
Row-level vs Schema-level vs Database-level isolation。一旦上線後再改，代價極高（data migration + 全面 API 審查）。必須在寫第一行 DB migration 之前確定。
決策二（最緊急）：Credential 管理架構與 Secret Rotation 策略
IntegrationProfile 目前完全沒有 credential 設計，但所有外部連線都需要認證。 影響 Connector 介面、onboarding 流程、Security Review。不能等到實作第一個 Oracle Connector 時才臨時決定。
決策三：Schema Mapping 版本不可變性與 AnnotationSet Schema Pin 機制
決定了 canonical model 必須記錄哪些元數據（schema_version_ref 欄位），及 SchemaStore 的 immutability 設計。若不在設計階段確定，migration script 將難以撰寫。
